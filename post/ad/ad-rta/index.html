广告投放-RTA（三）

从业务场景、请求链路到工程实现，讲清楚 RTA 是什么、为什么需要它，以及它和普通广告投放有什么区别。

在广告投放里，很多人先接触到的是计划、定向、出价、归因这些概念。等再深入一点，就会遇到一个更偏工程、也更偏策略的问题：平台能不能在一次广告请求发生的瞬间，结合广告主自己的数据，再决定这次到底要不要投？

这就是 RTA 经常出现的地方。

很多人第一次看到这个缩写，会把它理解成一种“更高级的定向”或者“更实时的出价”。这种理解不算错，但还不够准确。RTA 真正重要的地方在于，它把广告平台的流量机会，和广告主侧或第三方的数据判断，在实时链路上连接起来了。

简单说，RTA 是让外部决策能力参与实时广告请求判断的一种机制。能够让客户将自有的数据能力实时作用在广告决策环节，从而达到媒体与客户共建和提升广告效果的能力。

RTA 到底是什么

RTA 通常可以理解为 Real Time API 或 Real-Time Advertising 场景中的实时决策接口能力。不同平台对缩写展开不一定完全一致，但核心思想大体相同：

当广告平台收到一次流量请求时，不只依赖平台内部规则和模型，还会实时调用外部服务，结合更多业务数据，决定是否投放、如何出价、该返回什么策略。

相关概念

RTA token

广告主在媒体侧注册后，会获得一个 token，也称为 RTA token。

媒体侧则需要根据广告主的 token，调用广告主的 RTA 接口，获取实时的广告决策响应。

RTA token 主要有以下两个作用：

  • 认证：为防止广告主的接口被其它媒体侧调用，确保广告主的流量安全，媒体侧需要根据广告主的 token，调用广告主的 RTA 接口。
  • 策略：广告主可以创建多个 RTA token，为每个 token 配置不同的出价策略，来影响广告的投放行为。

RTA 策略

在一个 RTA 策略上可以绑定多个「人群包」和「出价规则」。

QA

RTA 和广告计划有关系吗？

答：有，可以把 RTA 想象成广告计划的“智能过滤器”或“终审员”。即使你在广告后台（计划层级）设置了定向逻辑（如：性别男、20-30岁），系统在真正出价前，还是会先拿着这个用户去问一遍你的 RTA 接口。

oCPX 是你告诉系统：“我出 10 块，帮我在这个池子里找人。”

RTA 是你告诉系统：“别听你的，听我的，这人值 12 块，那人值 2 块。”

UBMAX 是你告诉平台：“我每天有 10 万块，这是我的视频，我要 1 万个激活，你自己看着办。”
